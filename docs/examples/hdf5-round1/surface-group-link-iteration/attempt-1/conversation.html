The harness targets the group open and link traversal attack surface by exercising the full API call chain a real HDF5 application would use:

  1. H5Fopen (RDONLY) — writes fuzz bytes to a temp file and opens it as an HDF5 file
  2. H5Gopen2("/", ...) — opens the root group, triggering B-tree/fractal-heap parsing
  3. H5Gget_info — reads group metadata (storage type: compact vs. dense, nlink count)
  4. H5Giterate — iterates via the legacy symbol-table / B-tree code path; callback calls H5Lget_info2 per link to probe compact→dense transitions
  5. H5Lvisit2 — recursive link visitation via the modern v2 B-tree / fractal-heap path, run with both H5_INDEX_NAME and H5_INDEX_CRT_ORDER to cover both ordering code paths
  6. H5Lget_info_by_idx2 — queries link info by position index when links exist
  7. A second H5Gopen2("group", ...) pass exercises named non-root groups
  8. H5Gclose / H5Fclose on every opened identifier

The harness uses H5F_ACC_RDONLY (no writes to the file) and suppresses HDF5 error output with H5Eset_auto2(NULL, NULL) to keep fuzzer output clean. Callbacks always return 0 to continue iteration, never recurse beyond what the API itself controls, and touch no harness-owned buffers unsafely.
