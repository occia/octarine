Write a fuzzing harness for ONE specific attack surface of hdf5 (c, address).

Attack surface: Committed datatype open and introspection
Target API(s): H5Topen2, H5Tget_class, H5Tget_size, H5Tget_nmembers, H5Tget_member_type, H5Tget_member_offset, H5Tis_variable_str, H5Tclose
Why this surface: Committed (named) datatypes are stored in the file and decoded on H5Topen2. Compound, enum, vlen, and array type descriptors are deeply nested and hand-parsed; historically a rich source of OOB/UAF bugs.

This is a follow-up attempt. Feedback on the previous harness:
Replace the golden-image overlay approach with one that feeds fuzzer bytes directly as raw HDF5 file content (similar to h5_read_fuzzer.c): copy fuzz data into an in-memory buffer, open it via H5Pset_file_image + CORE VFD, then call H5Topen2 for a set of candidate names. This eliminates the hard H5Fopen gate caused by heavy mutation. If the golden-image approach is kept, reduce SAFE_OFFSET from 512 to 8 (just the HDF5 magic "\x89HDF\r\n\x1a\n") so the fuzzer can mutate the superblock and object-header regions where committed type descriptors are actually encoded; the current 512-byte protection zone prevents the fuzzer from reaching the datatype-parsing code at all. Also avoid silently truncating inputs longer than the golden image — instead allocate max(golden_sz, size + SAFE_OFFSET) to allow the fuzzer to explore larger type descriptor payloads.

Address it: if it was a false-positive crash, fix the API misuse; if the surface was not explored
thoroughly enough, improve how the harness drives these APIs.

Follow CLAUDE.md for the full workflow (download, write into fuzz-proj, build green via build-project,
write /work/agent/new-harnesses.txt). Exercise the API the way a real caller would — a crash from
harness misuse is a false positive and will be rejected.
