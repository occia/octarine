Attack surface: Committed datatype open and introspection
Target API(s): H5Topen2, H5Tget_class, H5Tget_size, H5Tget_nmembers, H5Tget_member_type, H5Tget_member_offset, H5Tis_variable_str, H5Tclose
Why this surface: Committed (named) datatypes are stored in the file and decoded on H5Topen2. Compound, enum, vlen, and array type descriptors are deeply nested and hand-parsed; historically a rich source of OOB/UAF bugs.

This is a follow-up attempt. Feedback on the previous harness:
The harness must overcome the file-format barrier so the fuzzer can actually reach H5Topen2 and the type introspection APIs. Two concrete fixes: (1) Construct a valid minimal HDF5 file in-memory at startup (using H5Fcreate into a memmap or tmpfs), commit several committed types of varying classes (compound, enum, vlen, array, string) to it, close and re-open it read-only, then use fuzz data to mutate bytes in the raw file buffer corresponding to the type-descriptor region only — so every input reaches a parseable file with at least one named type. (2) Alternatively, keep the current whole-file approach but supply a seed corpus of valid HDF5 files that each contain at least one committed datatype with compound/enum/vlen/array nesting. Without one of these changes the fuzzer never exits the superblock-parsing layer and the H5T* attack surface is never exercised.

thoroughly enough, improve how the harness drives these APIs.
